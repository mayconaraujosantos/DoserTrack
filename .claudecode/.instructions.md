# Doser Custom Development Rules

## Core Principles

1. **Tests First**: Write tests BEFORE implementation, always
2. **Quality Over Speed**: Take time to think through each step
3. **Never Skip Planning**: Plan thoroughly before coding
4. **Type Safety**: Strict TypeScript, no `any` types
5. **Error Handling**: Explicit error handling in all async operations

## Code Organization Rules

### File Structure

- `/app` - Screens and routes (Expo Router)
- `/components` - Reusable UI components
- `/lib` - Business logic, utilities, API integrations
- `/hooks` - Custom React hooks
- `/types` - TypeScript type definitions
- `/constants` - App constants and theme

### Naming Conventions

- Files: kebab-case (e.g., `medicine-service.ts`)
- Components: PascalCase (e.g., `MedicineCard.tsx`)
- Hooks: camelCase with `use` prefix (e.g., `useThemeColor`)
- Functions/variables: camelCase (e.g., `calculateNextDose`)
- Types/Interfaces: PascalCase (e.g., `Medicine`, `DoseSchedule`)

## TypeScript Rules

- No implicit `any` types - always explicit
- Use strict mode in tsconfig.json
- Prefer interfaces for contracts, types for unions/tuples
- Export types from types/index.ts

## Database Rules (SQLite)

- All queries should be async functions in `lib/database.ts`
- Use parameterized queries to prevent SQL injection
- Always handle database errors explicitly
- Use transactions for multi-step operations
- Profile filtering: ALL queries must filter by `profileId`

## State Management

- Use React Query for server/sync state (`lib/query-client.ts`)
- Use Zustand store for local UI state (`lib/store.ts`)
- Keep state normalized and colocated near components
- Avoid prop drilling - use context when necessary

## Authentication

- All auth logic in `lib/auth.ts` and `lib/biometrics.ts`
- Secure tokens in `expo-secure-store`
- Handle auth errors gracefully
- Implement session timeout

## API & Supabase

- All Supabase queries in `lib/supabase.ts`
- Implement offline-first sync patterns
- Handle network errors with user feedback
- Use real-time subscriptions for live data
- Conflict resolution strategy: last-write-wins with timestamps

## Performance

- Lazy load screens and heavy components
- Memoize components with `React.memo` when appropriate
- Optimize images with `expo-image`
- Use `useCallback` for memoized functions
- Profile bundle size regularly

## Error Handling

- All async/await wrapped in try-catch
- User-friendly error messages in UI
- Console errors for debugging (dev only)
- Error boundaries in critical sections
- Proper error recovery flows

## Testing

- Test structure: `__tests__` folders parallel to source
- Test files: `*.test.ts` or `*.test.tsx`
- Mock external dependencies (Supabase, SQLite, etc)
- Aim for >80% code coverage
- Integration tests for critical flows

## Notifications & Permissions

- Request permissions explicitly
- Handle denied permissions gracefully
- Clear notification documentation
- Test on real devices

## Accessibility

- WCAG 2.1 AA compliance
- Semantic HTML/components
- Proper color contrast
- Test with screen readers

## Git & Versioning

- Feature branches: `feature/name`
- Bug fixes: `fix/name`
- Commits: Present tense, descriptive messages
- PRs required for main branch

## Environment Variables

- `.env.example` must exist with all required vars
- Never commit `.env`
- Document all environment variables
- Validate required vars on app startup

## Security

- No sensitive data in code
- Sanitize user inputs
- Use HTTPS only
- Validate all API responses
- Keep dependencies updated

## Documentation

- JSDoc comments for public functions
- README for each major feature
- ARCHITECTURE.md for system design
- Inline comments for complex logic

## Debugging

- Use React DevTools when possible
- Leverage console for debugging
- Use breakpoints in VS Code
- Log important state changes
- Clean up debug logs before commit

## Build & Deployment

- Test locally on Android and iOS before deploy
- Use EAS for builds
- Version following semver
- Tag releases in Git
- Maintain changelog

## Performance Budgets

- App bundle size: < 5MB (iOS), < 10MB (Android)
- First load: < 3 seconds
- Navigation transition: < 500ms
- Database query: < 1 second

## Quality Gates

- All tests passing
- No console errors in production
- No TypeScript errors
- ESLint clean (configured in eslint.config.js)
- Code review approved
